# Phase 5.1: Tune OpenCodeAgent for Multi-Step Tool Execution - Context

**Gathered:** 2026-03-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Make a small local LLM reliably drive opencode's multi-step agent workflow for the superlint_demo task. Phase 5 Plan 03 proved the OpenCodeAgent infrastructure works (config injection, Docker detection, SIGSEGV retry, model unload) but the model cannot complete the 3-step task through opencode's 10+ tool protocol. This phase researches and tunes model selection, prompt engineering, Ollama parameters, and opencode configuration to achieve reward >= 0.85 from the deterministic grader. OpenCodeAgent is a must-have -- there is no fallback. Does NOT include CI setup (Phase 6) or performance comparison (Phase 7).

</domain>

<decisions>
## Implementation Decisions

### Go/no-go criteria
- Minimum reward threshold: 0.85-1.00 from deterministic grader; LLM grader handles remaining quality assessment
- OpenCodeAgent is a **must-have** -- no fallback plan; re-assess the situation if all options exhausted
- No hard time limit on experiments -- use judgment, thoroughness over speed
- Structured benchmark data for every experiment: markdown table with model, config, duration, reward, tool calls, notes (same rigor as Phase 4.1)
- 1-trial screening for each experiment; 3-trial validation on promising candidates only
- 1 warmup run before measured trials (consistent with Phase 4.1 methodology)

### Research priorities (before experiments)
- **High priority:** opencode internals research
  - Can opencode's tool set be restricted via config? (e.g., disable Skill, Plan, Question tools) -- root cause was 10+ tool overwhelm
  - What system prompt does opencode inject internally? Critical for crafting complementary prefix
  - All opencode.json config options investigated (permissions, doom_loop, model settings, etc.)
  - opencode docs/issues for known working models and model compatibility
  - How to trace Ollama API requests through opencode (OLLAMA_DEBUG, --verbose, HTTP proxy)
  - Check if opencode linux-arm64 SIGABRT issue #13367 is fixed in current release
- **High priority:** small-model agentic patterns
  - How SWE-Bench, Aider, OpenHands, and similar frameworks prompt small models for multi-step tool use
  - Tool-calling benchmark data across model families (Qwen, Phi-4-mini, Llama 3.1, Mistral, DeepSeek) -- include recent releases
  - Whether thinking mode helps or hurts small-model tool agents (research, then verify experimentally)
- **Medium priority:** 8B model resource management
  - OLLAMA_NUM_THREAD + OS-level priority (nice/process priority) for 8B models on ARM64 (qwen3:8b hogged the entire OS at 8 threads)
  - Claude determines right resource allocation based on research findings

### Model strategy
- Maximum model size: 8B (must fit on 16 GB CI runner sequentially with grader)
- Start with qwen3.5:4b (already closest to working -- completed step 1/3), build up systematically
- Explore the full experiment space: quantizations, Modelfile parameters, Ollama env vars, thread counts, thinking mode on/off/budget, other model families
- Research which model families are best for tool calling at 3-8B scale, including Phi-4-mini, Llama 3.1, Mistral, and recent releases
- Qwen family history: 2.5 good for grader, 3 promising for OllamaAgent, 3.5 newest and best tool-calling training

### Prompt & tool steering
- Prompt strategy determined by research (how proven frameworks handle small-model multi-step tool use)
- Single `opencode run` must complete the full multi-step task -- no sub-task decomposition (tests true agentic capability)
- Task instruction.md stays unchanged; OpenCodeAgent prefix wrapper is the tuning surface
- opencode.json config is a tuning variable (permissions, model settings, all options)
- Thinking mode (/no_think vs /think vs thinking_budget) included in experiment matrix after research informs which to try

### Platform requirements
- Local provider must achieve 0.85+ reward first
- Docker provider must also pass before Phase 5.1 closes
- Docker: test Linux ARM64 opencode build (native on WSL2-backed Docker Desktop, no QEMU emulation) -- verify SIGABRT issue #13367 status
- Local: keep existing SIGSEGV retry loop (3 retries for x64 QEMU instability on Windows ARM64)
- Experiment method: CLI with `--agent=opencode`, not a custom benchmark script

### Claude's Discretion
- Experiment sequence and prioritization within the research-informed search space
- Ollama parameter values (num_ctx, num_predict, num_batch, temperature, etc.) based on research
- Modelfile contents for each experiment
- Prompt prefix wording (informed by opencode's internal system prompt research)
- When to escalate from smaller to larger models

<specifics>
## Specific Ideas

- qwen3.5:4b was closest to working in Phase 5 Plan 03: first tool call succeeded, but model summarized after step 1 instead of continuing to steps 2-3. Classic small-model early-stopping behavior.
- qwen3:8b hogged the entire OS when tested in Phase 4 -- any 8B model needs careful thread/priority management
- opencode exposes 10+ tools via OpenAI-compatible /v1 API (bash, read, edit, glob, grep, list, Skill, question, plan...) while OllamaToolAgent only exposes 3. Tool count reduction via opencode config would directly address the root cause.
- Docker runs WSL2-backed on this machine, so Linux ARM64 containers run natively -- the Linux ARM64 opencode build should avoid all QEMU overhead if the SIGABRT issue is resolved
- No Windows ARM64 build of opencode exists; local runs use x64 under QEMU emulation with ~60% SIGSEGV rate (handled by retry loop)

</specifics>
