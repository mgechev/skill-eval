<decisions>
## Implementation Decisions

### Model exploration strategy
- Optimize qwen3:4b first before trying alternative models
- If qwen3:4b tuning falls short, Claude picks 2-3 fallback models based on online research
- Native tool calling preferred; text-based tool call parsing acceptable as last resort only
- Model name must reflect the base model (e.g., `qwen3-skill-eval-agent`, `qwen2.5-skill-eval-agent`)
- Try 1-2 quantization variants (e.g., q4_0 vs default Q4_K_M) to measure speed/quality trade-off
- Prefer CI-safe models (agent + grader must fit sequentially on 16GB), but don't exclude a clearly superior local-only model
- Minimum reward threshold: 0.90+ required (current baseline: 0.97)
- Quick NPU investigation: researcher checks if any practical Ollama NPU path exists for Snapdragon X Elite; defer if no easy win

### Benchmarking methodology
- superlint_demo task only, 3 trials via existing `--trials=3` CLI flag
- 5-minute target applies to full trial wall-clock time (agent execution + model unload + grading)
- Capture full Ollama per-turn metrics: eval_count, eval_duration, prompt_eval_count, prompt_eval_duration, load_duration
- Separate benchmark script -- do not modify OllamaToolAgent internals for metric capture
- Benchmark results stored as phase-level documents in `.planning/phases/04.1-*/` (like v1.0 Phase 2.1 benchmark-results)
- Warm-up run before measured trials to eliminate cold-start noise
- Report variance (standard deviation) across trials but no hard threshold -- use judgment
- Benchmark config input approach: planner decides (CLI args vs config matrix file)
- If no config hits 5-min target: escalate to code-level optimizations (context pruning, parallel tool execution, etc.)

### Prompt engineering approach
- System prompt only -- do not modify task instruction files
- Research-driven rewrite: consult https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents and https://developers.openai.com/blog/eval-skills for recommended agent settings
- "Research" in this phase means online research for best practices and model-specific techniques, followed by trial and error in the pipeline
- Research few-shot examples: investigate token cost vs turn reduction for small-model tool agents
- Research context pruning strategies: trimming older tool results from conversation history to keep context small in later turns
- Research Modelfile SYSTEM directive vs API system message: check previous research artifacts + online sources, then trial and error
- maxIterations: research optimal value from actual turn count data (current 30 was a guess; test file uses 10)
- maxToolOutputChars: research from online sources (SWE-Bench, Aider, OpenHands patterns) and local run data

### Ollama parameter space
- Agent Modelfile must be tuned independently from grader -- different workloads need different settings (temperature 0 was for grader determinism, not agent creativity)
- OLLAMA_FLASH_ATTENTION and OLLAMA_KV_CACHE_TYPE: include in experiment matrix, NOT applied blindly (known to slow down local inference on Snapdragon while helping CI)
- num_ctx and num_predict: include in experiment matrix (test whether /no_think allows lower num_predict)
- Temperature: researcher determines optimal for agent workloads based on eval articles and model-specific docs
- Keep 8 threads locally (not re-testing)
- Researcher recommends which additional Ollama parameters (num_batch, repeat_penalty, top_k, top_p, mirostat) to experiment with based on online research for agentic tool-calling workloads
- keep_alive strategy: research best practice for benchmarking vs production
- OLLAMA_NUM_PARALLEL: include in research (verify serial agent workload behavior)
- Streaming (stream: true vs false): research for tool agents, considering wall-clock time metric
- Ollama version: assume latest, don't introduce as a variable

### Claude's Discretion
- Tool description optimization (shorter descriptions to reduce per-turn tokens)
- Custom TEMPLATE directive in Modelfile for tool call formatting
- Thinking suppression overhead evaluation (non-thinking models vs /no_think on thinking models)
- Benchmark config input format (CLI args vs matrix file)

<specifics>
## Specific Ideas

- v1.0 Phase 2 research (MODELS-AND-OPTIMIZATIONS.md) documented OLLAMA_FLASH_ATTENTION, OLLAMA_KV_CACHE_TYPE, num_batch, and OLLAMA_NUM_PARALLEL as "no quality trade-off" optimizations -- but user reports flash attention and KV cache quantization slowed local inference on Snapdragon X Elite. Must benchmark, not assume.
- Phase 4 found that an aggressive "ONLY tool calls" system prompt caused 18-command loops. Current directive prompt works but may not be optimal.
- Phase 4 benchmarks: single turn 48.9-76.5s, full eval (9 turns) 440-590s. Target is 300s average.
- Eval articles to consult for agent temperature and prompt recommendations:
  - https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents (skill-eval inspiration)
  - https://developers.openai.com/blog/eval-skills (skill-eval inspiration)
